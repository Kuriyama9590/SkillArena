# Skill 自改进示例

> 本文展示 `improve_skill` 在两种典型场景下的"改进前 vs 改进后"对比。
> 所有 v2 文本都是"合理的预期结果",不需要真实跑 API。

---

## 示例 1 · concise-writer · v1 → v2(纠"骨架感")

### 弱点列表(从最近 20 场 Elo 对战评判中聚合)

1. **缺例子**:平均 correctness 6.8,频繁被评判标记"虽然简洁但没具体例子,读者抓不住"。
2. **冷冰冰**:creativity 6.2,被标记"风格太冷,缺人情味,长文场景下读者疲劳"。
3. **句式单调**:clarity 7.5 但被标"全是一句话一段,缺少节奏感"。

### v1(原 skill)

```markdown
# Concise Writer · 简洁写作风格

## 核心原则

1. 直接进入主题:第一句话就要点明观点或答案,不要铺垫背景。
2. 一段一意:每段只表达一个核心想法,避免一锅炖。
3. 短句优先:能 20 字说清的不写 50 字。
4. 删掉一切冗词:删掉"非常""其实""一般来说"这种填充词。
5. 结尾收束:最后一句给出结论、行动或核心金句。

## 不允许的写法

- 开头"在当今社会..."、"随着时代发展..." 等套话。
- 反复复述同一论点。
- 用三句话解释一个能一句话说清的概念。

## 输出格式

- 总字数比用户要求的**少 20%~30%** 优先。
- 段落之间空一行。
- 如有需要,使用编号列表而不是段落。
```

### v2(自改进后)

```markdown
# Concise Writer · 简洁写作风格

## 核心原则

1. **开门见山**:第一句话点明观点或答案,先给"是什么"。
2. **一段一意**:每段只表达一个核心想法,关键论点必须配一个具体例子(数字、场景或引语)。
3. **短句节奏**:能 20 字说清的不写 50 字;长段落拆成"短句 + 短句"节奏,避免单调。
4. **删冗词留人味**:删掉"非常""其实""一般来说"等填充词;
   但保留必要的人情味过渡(例如"换句话说""这意味着"),不让语气太冷。
5. **结尾给动作**:最后一句给结论、行动或核心金句,不要意犹未尽地"总结一下"。

## 行为约束

- 禁止"在当今社会""随着时代发展"等套话开头。
- 禁止反复复述同一论点;每段一个新想法。
- 禁止用"比如某些情况下"这种空泛例子 —— 例子必须具体到场景或数字。
- 禁止连续 5 段以上都是单句,要有节奏变化。

## 示例

输入:用 100 字介绍单元测试。
输出:
单元测试是对单个函数或模块的自动化验证。
例如对 `add(a, b)` 写 3 个测试:正常输入返回正确值、负数处理、类型错误抛异常。
每次改完代码跑一遍,几秒内就能发现"我刚刚把加法写成了减法"这种低级错误。
```

### 改进效果点评

- **针对 weakness 1(缺例子)**:在原则 2 中显式要求"关键论点配例子",行为约束新增"禁止空泛例子"。
- **针对 weakness 2(冷冰冰)**:原则 4 明确"保留必要的人情味过渡",给出具体可用短语。
- **针对 weakness 3(句式单调)**:原则 3 强调"短句 + 节奏",行为约束新增"禁止连续 5 段单句"。
- **保留原 skill 的"简洁 + 短句 + 删冗词"内核**,未批评的部分(开门见山、结尾收束)被原样保留。

## 示例 2 · detailed-writer · v1 → v2(纠"啰嗦")

### 弱点列表

1. **铺陈过多**:completeness 9.2 但 creativity 5.8,评判"虽然全但太啰嗦,关键结论被淹没"。
2. **例子有时空泛**:被多次标记"比如某些情况下"这种无意义例子。
3. **结尾偏弱**:clarity 7.0,常被"详尽但结尾没收住,像没写完"。

### v1(原 skill)

```markdown
# Detailed Writer · 详细写作风格

## 核心原则

1. 充分铺垫:在给出结论前,先交代背景、动机、相关概念。
2. 举例支撑:每提出一个论点,**至少给一个具体例子**。
3. 多角度展开:从利弊、对比、历史、当下、未来等至少 2 个角度。
4. 层次推进:"是什么 → 为什么 → 怎么做 → 注意事项"。
5. 解释专有名词:遇到专业术语时,用一句话解释。

## 不允许的写法

- 只给结论不给理由。
- 例子是"比如某些情况下"这种空泛表达。
- 把所有内容塞进一段。

## 输出格式

- 默认总字数比用户要求**多 30%~50%**。
- 每个主要段落之间用空行分隔。
- 优先使用"小标题 + 段落"的结构。
```

### v2(自改进后)

```markdown
# Detailed Writer · 详尽但收得住的写作风格

## 核心原则

1. **铺垫有度**:用 1-2 句交代背景和动机即可,不要超过全文 20%;
   关键结论要在前 1/3 处出现,不要藏到最后。
2. **例子必须具体**:每论点至少配一个具体例子(数字、场景、引语),
   避免"比如某些情况下""一般来说"这种空泛表达。
3. **多角度收口**:从利弊、对比、历史、当下、未来中选 2-3 个角度;
   每个角度 1 段,不要在同一段堆砌 4 个角度。
4. **四段式推进**:"是什么 → 为什么 → 怎么做 → 注意事项",
   每段不超 150 字。
5. **解释术语**:遇到专业术语时,用一句话解释,放在第一次出现的位置。

## 行为约束

- 禁止只给结论不给理由;但也禁止"理由 + 理由 + 理由"无限堆砌。
- 禁止空泛例子(例如"某些情况下""大多数时候")。
- 禁止"小标题 + 长段平铺"——每个小标题下必须有 2 段以上。
- 禁止"开放结尾"——必须以"行动 / 注意事项 / 一句话总结"收口。

## 示例

输入:解释微服务架构的优缺点。
输出:
微服务是把一个大应用拆成多个独立小服务,每个服务自己部署、自己扩缩容。
优点:某服务挂了不影响整体,例如推荐服务挂了,主站仍可下单;
团队可以独立迭代,A 团队改搜索、B 团队改支付,互不阻塞。
缺点:跨服务调用变多,一次"下单"可能要走 6-7 个服务,排查问题时日志分散在多个系统。
建议:团队规模 < 20 人时优先单体,别为"未来"提前付出微服务的复杂度成本。
```

### 改进效果点评

- **针对 weakness 1(铺陈过多)**:原则 1 显式约束"铺垫不超过 20%,关键结论在前 1/3"。
- **针对 weakness 2(空泛例子)**:原则 2 和行为约束明确"禁止某些情况下"等措辞。
- **针对 weakness 3(结尾偏弱)**:新增行为约束"禁止开放结尾",原则 1 强调"关键结论在前 1/3"。
- **保留原 skill 的"例子支撑 + 多角度 + 解释术语"**,只调整比例和收尾。

## 调用方式(节选自 `deliverable-fusion-engine.md`)

```python
from arena.self_improve import run_improvement_cycle

report = run_improvement_cycle(
    skill_name="concise-writer",
    skill_content=Path("skills/concise-writer.md").read_text(encoding="utf-8"),
    max_iterations=3,
    target_elo_delta=20.0,
    evaluator=my_evaluator,  # 注入真实 evaluator(由 orchestrator 提供)
)
print("converged:", report.converged)
print("final_elo:", report.final_elo)
print("steps:", len(report.steps))
for step in report.steps:
    print(f"  iter={step.iteration}  Elo {step.elo_before:.0f} -> {step.elo_after:.0f}  Δ={step.elo_delta:+.1f}")
```
