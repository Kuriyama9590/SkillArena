### 0.3 术语

| 术语 | 含义 |
|---|---|
| 赛道（track） | 一类能力的竞技场，任务和 skill 都向赛道归属，同赛道才对战 |
| skill | 一段 system prompt（`.md`），引导执行模型生成产物；竞技台的"选手" |
| general | 跨域属性（非赛道），声明 general 的 skill 报名全部赛道 |
| baseline | 裸 prompt（不加任何 skill）的虚拟选手，通吃所有赛道，作为参照零点 |
| 任务（task） | 赛道里的"题目"，由 prompt 构成，创建时即归属某赛道 |
| 对战（match） | 同赛道两个选手在同一任务上的两两比较，产出胜者与 Elo 变动 |
| 产物（artifact） | 融合或自改进生成的新 skill，需过定级赛才可能入榜 |
| 定级赛（placement） | 新 skill / 产物入榜前的 1-2 场校准对战 |

## 1. 产品定位与用户

### 1.1 一句话定位

把"prompt skill（系统提示词工程）"变成可量化、可对比、可累积的资产：在同一批赛道任务上，让多个 skill 引导同一执行模型生成产物，由评判模型盲评，用分赛道 Elo 累积排名。

### 1.2 用户与使用姿态

- **主用户**：prompt 工程师 / skill 作者。通过 Web 控制台启动竞技、实时观看对战与流式输出、查看天梯、管理 skill、回放历史 run。
- **使用入口**：浏览器。安装方式为 `uvicorn` 起后端 + `npm run dev` 起前端（详见 README 重写后的"快速开始"）。
- **核心体验**：实时直播式观看（流式输出 + 实时打分 + 实时排名），不是跑完看报告的批处理工具。

### 1.3 核心假设

- skill 是可对比、可迭代的资产。
- 对比必须基于统一赛道、统一评判维度、统一匿名协议，否则排名噪声极大。
- Elo 跨多次 run 累积才有意义（见 §8.2）。

## 2. 赛道模型

### 2.1 赛道清单（6 条专用 + 1 跨域属性）

**选赛道判据**：只开"当前 LLM 能力分化大"的领域（信号优先），不做"能力图谱铺满"。凡是现代 LLM 普遍做得好的能力（翻译、普通总结、基础改写）一律不开——比出来全是平局，Elo 无信号。

| # | 中文名 | 标识符（代码用） | 边界（判断归属的标准） |
|---|---|---|---|
| 1 | 代码 | `coding` | 产出可执行代码或技术方案 |
| 2 | 写作 | `writing` | 产出有立场/有原创表达的成文（议论文、文案、解释性长文、文学翻译这类再创作） |
| 3 | 推理 | `reasoning` | 需要分析、综合、判断、多步推导（无明确标准答案） |
| 4 | 角色扮演 | `roleplay` | 扮演特定身份/人格完成任务 |
| 5 | 指令遵循 | `instruction` | 复杂多约束指令全满足（含结构化输出 / schema 约束等子类） |
| 6 | 长文本处理 | `longtext` | 超长输入下检索/定位/综合，"长"带来的遗忘/遗漏是核心考察点 |

- **`general`**：跨域属性（非赛道）。声明 `general` 的 skill 报名全部 6 条赛道。
- **排除的候选及理由**：翻译（LLM 普遍做得好，无信号）/ 总结（信号弱）/ 数学（与推理重叠，skill 调节空间小）/ 数据提取（与推理重叠）/ 改写润色（与写作重叠）/ 对话客服（任务难造）/ 创意故事（与写作重叠且评判主观）/ 结构化输出（并入指令遵循，为其子集）/ Agent（复杂且市面饱和）/ 审阅·教学（不要）。

### 2.2 赛道是独立维度

赛道先于 skill 和任务存在。**任务在创建时即归属某赛道**（像考试题天然属于某科），skill 用 `domains` 声明自己报名哪些赛道。配对规则：**任务赛道 ∈ 选手报名赛道集合** 时才同台对战。

> 与现状的差异：现 `tasks/fixed/*.yaml` 用 `category` 字段（writing/coding/analysis）驱动赛道，且 `TASK_DOMAINS` 仅 3 条。需扩展为 6 条赛道，旧 3 类任务逐条重新归类（`analysis` → `reasoning` 或其他，逐条判断），并为 reasoning/roleplay/instruction/longtext 补题。

### 2.3 严格一对一

非 general skill **严格一对一**：一个 skill 只属 1 条专用赛道。代码现状已支持该约束的两个机制：`_validate_skill_domains`（`backend/routers/arena.py:68-91`，API 层拒绝跨专用领域选择）与 `_domain_is_active`（`arena/orchestrator.py:1358`，编排层赛道激活锚定——存在专用 skill 时某赛道必须有专用 skill 锚定，general 不能独自开赛道）。两者语义保留，赛道词替换为新的 6 条。

> 注：grill 中曾讨论"主+辅/交集兼容域"模型，**已被否决**，坚持严格一对一。

### 2.4 动态开榜

某专用赛道 **≥2 个专用 skill** 才立该赛道榜。不足时该域 skill 暂只与 general/baseline 对战、不单独立榜（防空天梯）。这是 6 赛道下避免空榜的唯一手段。

### 2.5 赛道立起来的前置条件（内容建设）

每条赛道要"自养"，须同时满足：
1. ≥2 个专用 skill
2. ≥5 条同域任务（供 Elo 有足够样本）
3. 赛道边界足够清晰，skill 与任务能稳定归入

长文本赛道的唯一难点是素材，用四大名著等公版长文本兜底。其余赛道任务易造。

## 3. 选手模型

### 3.1 三类选手

| 类型 | system prompt | 报名赛道 | 是否进天梯 |
|---|---|---|---|
| 专用 skill | 有，针对某赛道 | 其声明的 1 条赛道 | 过定级赛后入榜 |
| general skill | 有，通用 | 全部 6 条赛道 | 过定级赛后入榜，每条赛道独立 Elo |
| baseline | 无（裸 prompt） | 全部 6 条赛道 | **必须出现且争冠** |

### 3.2 baseline 语义

- baseline = 不加任何 skill 的虚拟选手（现状实现：`skill_content=None`，不设 system message，仅 user message 放 task prompt）。
- baseline 是**系统内置参照**，初始 1500 直接在榜，**不过定级赛**（定级赛只针对新 skill / 产物）。
- **baseline 必须出现在每条已开榜赛道的榜上，且参与排名争冠**（与现状冲突：现前端 `Leaderboard.tsx:35` 过滤掉 `baseline` 开头项，不展示；需改为展示并参与排名）。未开榜赛道本就无榜，baseline 在其上无榜可争。
- baseline 通吃所有赛道，每条赛道一个独立 Elo。
- baseline 作为"无 skill 参照零点"：它的排名直观体现"加 skill 到底有没有用"。若 baseline 长期位居榜首，说明该赛道 skill 整体无效——这是有价值的信号，不该被隐藏。

### 3.3 general 的多榜问题

general skill 在每条赛道都有独立 Elo 分，进每条赛道榜。**无跨赛道总榜**，因此 general 在 6 条赛道榜上各出现一次，属正常。

> 前端提供"显示/隐藏 general"勾选，避免 general 在每条榜都出现造成视觉重复。

## 4. 排名模型

### 4.1 无跨赛道总榜

**不设跨赛道统一排名**。理由：不同赛道的 Elo 分布松紧不同，绝对分不可比（`coding` 的 1500 ≠ `writing` 的 1500）。

### 4.2 首页 = 6 冠军展示墙

首页（Dashboard）核心是 **6 条赛道各自冠军的展示墙**——陈列各赛道分冠军，**不可比、不跨赛道排序**。展示墙是"奖杯陈列"，不是"统一排名"。

> grill 中曾提出"按领先赛道均值的程度做跨赛道可比总榜"，**已否决**，改为不可比的冠军展示墙。

### 4.3 分赛道榜

每条赛道独立天梯：按 Elo 降序排列该赛道所有入榜选手（含 general 与 baseline）。general 可勾选隐藏；baseline 必显且争冠。

### 4.4 动态开榜（重申）

不足 2 个专用 skill 的赛道不单独立榜，该域 skill 只与 general/baseline 对战、其结果不展示为独立天梯。

## 5. 竞技机制

### 5.1 配对

- 同一赛道内，所有报名该赛道的选手（专用 skill + general + baseline）两两配对（`combinations`）。
- 每对选手在该赛道的每条任务上跑 `rounds_per_pair` 轮。
- 总场数 = Σ(赛道) [ 任务数 × 配对数 × rounds_per_pair ]。现状公式已在 FIX_PLAN BUG13 修正（乘了任务数）。

### 5.2 执行

- 每个 skill 的完整内容作为 system message 注入执行模型；baseline 不设 system message。
- 执行模型支持**流式输出**，token 通过 SSE 事件 `skill_output_start` / `skill_output_chunk` / `skill_output_done` 实时推给前端（现状已实现：`orchestrator.py:788-822` + `useArenaStatus.ts:113-122`）。

### 5.3 评判（盲评）

- 评判模型对两段**匿名**产物（标注 `Response A` / `Response B`，不泄露 skill 名称）做盲评。
- 评判维度（现状 `JUDGE_DIMENSIONS`）：`correctness` / `completeness` / `clarity` / `creativity`，各 0-10 分。
- 评判输出 `Verdict`：`winner`（A/B/tie）+ 双方维度分 + `reasoning`（评委评语）。

### 5.4 Elo

- 标准 Elo：K=32，初始分 1500。
- **分赛道分仓存储**（现状已实现 `domain_elo`：`{domain: {skill: rating}}`），每个选手在每条赛道有独立 Elo。
- **Elo 只由 `winner` 驱动**：`to_score()` 将 winner 映射为 1.0/0.5/0.0，维度分（correctness 等）**不参与 Elo**，仅作展示与改进输入信号。
- 总分守恒：一场对战 ΔR_A = -ΔR_B。
- baseline 作为普通选手参与 Elo 运算（初始 1500），无特殊逻辑。

> 设计要点：维度分不进 Elo 是有意为之——维度分供"为什么赢/输"的归因（喂给改进），Elo 只反映"赢没赢"的累积战绩。需求文档明确这一分离，避免后续误把维度分加权进 Elo。

### 5.5 定级赛（placement）

所有新 skill（含手动新增、融合产物、自改进产物）入榜前必须打 **1-2 场定级赛**（vs baseline 或当前赛道中位选手）：
- 完成 → 拿到校准 Elo，正式入榜。
- 未完成 → 标"未入榜"，不进天梯。
- **定级赛必须保存比赛结果 + 评委评语**（评语是后续改进的输入信号，不可丢）。
- 定级赛对手选择：优先当前赛道中位选手；若该赛道尚未开榜（无其他专用 skill），退化为 vs baseline。
- **baseline 本身不过定级赛**（系统内置参照，直接 1500 在榜）。
- **general 的定级赛**：general 跨 6 赛道，**6 条赛道各打定级赛**（准确性优先，不省成本）。general 在每条赛道拿一个独立校准 Elo，分别进对应赛道榜。

> 现状无定级赛机制，新 skill 直接以 1500 入榜。需新增。

## 6. 产物与改进闭环

### 6.1 产物 = 独立新 skill

融合产物（`fuse(A,B)`）、自改进产物（`improve(A)`）、融合+改进产物（`improve(fuse(A,B))`）各自作为**独立新 skill** 入榜，拿独立 Elo，不覆盖父代。

**融合只在同赛道内进行**：阶段 B 取某赛道的 Top2（A、B 必属同一赛道），融合产物继承该赛道。因此 §6.4 比较 max(A,B) 时，A、B 的 Elo 是同一赛道下的，可比。

### 6.2 血缘标注

入榜产物在天梯上标注血缘，形如 `fuse(A,B)` / `improve(A)` / `improve(fuse(A,B))`。

### 6.3 只保留 3 类冠军产物形态

每次 run 产出多个中间产物，天梯上只保留 3 类"最好"的冠军产物：
1. 最好融合型
2. 最好迭代改进型
3. 最好融合+改进型

每次 run 各留该类冠军（累积，非全局只养 3 个）。

**"冠军产物"从入榜产物中选**：只有通过定级赛、超越原始版本、正式入榜的产物才有资格当选该类冠军（§12 列"最好"判定口径待定）。

### 6.4 入榜门槛 = 超越原始版本

产物过定级赛后，**定级 Elo 超越其原始版本**才入榜，否则判劣化：
- 改进产物 `improve(A)`：定级 Elo > A 才入榜。
- 融合产物 `fuse(A,B)`：定级 Elo > **max(A,B)**（较强父代）才入榜。门槛最高——融合必须 1+1>2 才有意义，否则直接用较强父代即可。

### 6.5 劣化产物处理

劣化产物（未超越原始版本）**不入榜，但重点归档，不丢弃**。归档须存：
- 产物本身（完整内容）
- 父代血缘（A、B 是谁）
- 定级赛结果（胜/负/比分）
- 评委评语（reasoning）
- 劣化点摘要

理由：**踩过的坑不能踩第二次**。劣化产物的评语是下次改进的负样本输入。

### 6.6 自改进循环语义修正

阶段 C（自改进）不再假设"改进必成功"。改进产物过定级赛，超越父代入榜，未超越归档为劣化。改进达 `max_improve_iterations` 上限仍未超越父代 → **彻底放弃 + 归档**（不降级入榜，不无限重试）。

> 现状 `run_improvement_cycle` 默认成功，需改为"定级赛判定"。产物现状落 `reports/improved/` 临时目录，需改为持久化归档 + 血缘结构。

### 6.7 改进的负反馈闭环

```
产物定级赛
  ├─ 超越原始版本 → 入榜
  └─ 未超越（劣化）→ 归档 + 存评委评语
        ↓
   下次改进时，把历史劣化产物的评语作为负样本注入改进 prompt
        ↓
   产出新版本，再过定级赛 ……
```

### 6.8 劣化记忆库（新组件，待设计）

持久化存储所有劣化产物的评语，作为改进的负样本输入。
- 待定：记忆保留多久（防膨胀）、检索方式（按赛道？按父代血缘？）、注入改进 prompt 的格式。
- 现状无此组件，需新建。先记 todo，后续展开设计。

## 7. skill 生命周期

### 7.1 创建

- 新 skill 通过 Skills 页创建/编辑。
- **创建时必须选择赛道**（严格一对一，单选 radio），或选 general。
- 现状 SkillEditor 无赛道选择 UI，靠用户在 textarea 手写 front matter（`domains: [writing]`）。需改为 UI 单选控件，由后端写入 front matter。

### 7.2 入榜

新 skill 经定级赛（§5.5）后入榜；未过定级赛标"未入榜"。

### 7.3 删除 = 软删除

删 skill 一律**软删除**（标 `deleted`/`inactive`），不物理删文件。
- 软删后：不参与新 run 配对，但天梯历史榜与回放仍可见（灰色标注"已停用"）。
- **血缘保护**：软删时若该 skill 有后代（融合/改进产物）在榜，后代血缘记录补存该 skill 内容快照，防血缘悬空。
- Elo 通缩：不管（单 skill 进退对全局均值影响 <5 分，可忽略）。

理由：保护历史回放、血缘链、可复现性，符合 §8"所有回放都能看"的总原则。

## 8. 状态与持久化契约

### 8.1 所有回放都能看

每一次 run 的事件流持久化为 `reports/events/run_<timestamp>.jsonl`。任何历史 run 都可回放（现状已实现：`GET /api/arena/runs` + `/runs/{filename}` + 前端 `loadReplay`）。

### 8.2 页面刷新维持状态

- 进程内运行态：通过 SSE 事件流 + `orchestrator_state.json` 维持。
- **进程重启后**：模块加载时 `_reconstruct_status_from_disk()` 回放最近 run 事件文件重建 `_active_status`（强制 `running=False`），并用 `elo_state.json` 恢复 Elo 快照（现状已实现）。
- Elo **跨 run 累积**：`elo_state.json` 是跨多次 run 的权威源，不随单次 run 重置。

> grill 中曾讨论"跑完即焚/不留历史"，**已否决**。不留历史会让 Elo 退化为单轮排名，K 因子、分仓、general 跨域等设计全部失去意义。

### 8.3 加新 skill 不需重跑历史

新 skill 加入不重跑历史对战。它经定级赛拿校准 Elo 后入榜，后续自然对战中收敛。Elo 守恒：新 skill 带 1500 入系统，不偷不抢。

### 9.1 导航（6 → 5 入口）

| 页面 | 路由 | 职责 |

| Skills | `/skills` | skill 管理：按赛道分组 + CRUD |
| Matches | `/matches` | 历史对战记录 |
| Reports | `/reports` | Markdown 报告归档 |

### 9.2 Dashboard（首页 = 天梯）

- **核心：6 冠军展示墙**——陈列 6 条赛道各自的分冠军（名称 + Elo + 赛道标识），不可比、不跨赛道排序。
- **概览数据**（展示墙上方或侧栏）：总 skill 数、各赛道 skill 数、最近一轮 run 摘要。
- general / baseline 在展示墙中的显示：baseline 必现且争冠（若某赛道 baseline 是冠军，如实展示）；general 不出现在"冠军"位（除非它真是该赛道冠军），但其存在性在分赛道榜可见。
- 现状：Dashboard 是 4 卡片概览 + 各领域冠军区域 + 最近比赛（`Dashboard.tsx`）。需从"概览"改造为"天梯首页"，强化 6 冠军展示墙。

### 9.3 Arena（竞技控制台）

#### 9.3.1 控制面板

- **赛道选择**（新增）：可选跑哪几条赛道（多选，默认全部）。现状无此控件，领域靠选 skill 隐含。
- skill 选择（多选，留空=全部报名该赛道的 skill）。
- 任务来源（fixed/auto/hybrid）。
- 每对轮数 `rounds_per_pair`、改进迭代数 `max_improve_iterations`。
- 阶段 B 融合 / 阶段 C 改进 开关。
- 运行 / 重置 按钮。

#### 9.3.2 运行中（实时直播）

三件事同时呈现：
1. **实时打分**：每场对战结束即显示 score_a/score_b + winner（现状 `phase_a_match` 事件已支持）。
2. **实时排名**：Elo 快照随对战更新（现状 `elo_snapshot` + `EloLeaderboard` 已支持）。
3. **流式输出**：各 skill 边生成边看的流式 token（现状 `skill_output_*` 事件 + `SkillOutputViewer` 已支持）。

#### 9.3.3 流式输出呈现

- **左右分屏**：当前对战 skill A vs skill B 的流式输出左右对照。
- **当前场 + 侧栏场次列表**：一次 run 多场对战，主区只显示当前进行的那场；当前场结束自动进下一场正在进行的；侧栏场次列表可点选切回已结束的历史场。
- 进度条 / 时间线（A/B/C/D 四阶段状态）/ 事件日志。

#### 9.3.4 回放

- 回放历史 run 时**只展示结果**，不重跑、不流式。
- 现状已实现回放模式（`loadReplay` / `clearReplay`，橙色"回放模式"徽章，禁用控件）。

### 9.4 Skills（按赛道分组）

- 按 6 赛道 + general 分组展示（现状按 `domains[0]` 分组，赛道词需替换）。
- 创建/编辑 skill 时**单选赛道**（严格一对一）或 general，由 UI 控件设置，后端写 front matter。
- 支持搜索、按赛道筛选。
- 软删除标注"已停用"。

### 9.5 Matches / Reports

- Matches：历史对战记录，可按赛道筛选，点开看单场详情（含维度分、评委评语、双方产物）。现状已实现。
- Reports：每次 run 生成的 Markdown 报告归档。现状已实现。
- 这两页是 Arena 的产物展示，保留。
